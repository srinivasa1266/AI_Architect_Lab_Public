# Inferring

Inferring means using an LLM to identify signals from text.

Examples include:
- sentiment
- urgency
- intent
- topic
- risk level
- category
- action needed

This is one of the most practical prompt-engineering patterns for enterprise work.

---

## Why inferring matters

A lot of enterprise data is unstructured:
- emails
- support tickets
- notes
- incident descriptions
- customer messages
- claim narratives

Inference turns that into structured signals.

---

## Example tasks

- classify ticket priority
- identify fraud indicators
- detect customer dissatisfaction
- route issues to the right team
- label operational risk level

---

## Example prompt

You are a support triage assistant.

Classify the following ticket as LOW, MEDIUM, or HIGH urgency.

Rules:
- HIGH: outage, payment failure, security issue
- MEDIUM: repeated failure, delayed delivery, customer escalation
- LOW: general inquiry or minor inconvenience

Return JSON with:
urgency, reason

Input:
"""
Customer reports order delayed by 3 days and threatens cancellation.
"""

---

## Why this works

- role is clear
- categories are defined
- decision boundary is explicit
- output is structured

---

## Common risks in inference prompts

- labels are too vague
- categories overlap
- no examples are provided
- output is not constrained
- model invents rationale unsupported by input

---

## Best practices

- define categories explicitly
- reduce overlap between categories
- provide examples if category boundaries are subtle
- validate results in code when possible

---

## Enterprise note

Inference outputs often drive actions:
- routing
- escalation
- prioritization
- reporting

So even if inference is generated by an LLM, the output should not be treated as final truth. It should often be validated against rules or workflow context.

That is where systems like Panovisor become relevant later.
